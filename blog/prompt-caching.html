What Prompt Caching Changed in Production Cost

Mwiko Sinyangwe · Nearby Vibes · Build notes

Prompt caching did not change the product architecture. It changed the economics of running it.

In Nearby Vibes, the expensive part of an agent call was not always the user's latest message. It was the repeated context around the message: the system prompt, tool definitions, behavioral rules, examples, and working context that barely changed between calls. Before caching, every request paid for that entire static prefix again from scratch.

After caching, the cost profile shifted from "pay for the whole prompt every time" to "pay to write the stable prefix once, then pay a much lower read price while it stays reusable."

What We Were Repeating

Nearby Vibes used Claude as a tool-using agent. Each request included a lot of stable material alongside the user's actual message:

  • The core system prompt and response style rules
  • Tool definitions and JSON schemas
  • Tool usage policy and behavioral constraints
  • Examples of good tool calls
  • Product-specific safety and fallback behavior

The actual user message was often tiny — "Anything good nearby that's still open?" — but the model also needed the full operating manual: how to search venues, how to rank results, how to ask follow-up questions, when to pull user memory, and how to format the response.

That operating manual was the obvious caching candidate.

What We Cached

The most useful cached prefix was: tool definitions, system instructions, stable examples, and any long-lived context that did not change per user request. I avoided caching highly dynamic state unless I could guarantee it would be reused across the next several calls.

Anthropic's prompt cache is prefix-based. The API considers content in this order:

tools → system → messages

That ordering matters. If your tools change between requests, everything below them in the cache is also invalidated. Tool definitions need to be stable in production.

Implementation

The implementation is straightforward — you mark the end of the stable prefix with cache_control:

import Anthropic from "@anthropic-ai/sdk";

const anthropic = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY,
});

// Mark the last tool in the stable set as the cache boundary
const tools = [
  {
    name: "search_nearby_places",
    description: "Search for nearby restaurants, cafes, bars, and activities.",
    input_schema: {
      type: "object",
      properties: {
        query:          { type: "string" },
        lat:            { type: "number" },
        lng:            { type: "number" },
        open_now:       { type: "boolean" },
        radius_meters:  { type: "integer" },
      },
      required: ["query", "lat", "lng"],
    },
  },
  {
    name: "get_user_memory",
    description: "Retrieve relevant user preferences and past interactions.",
    input_schema: {
      type: "object",
      properties: {
        query: { type: "string" },
      },
      required: ["query"],
    },
    // Cache boundary: everything above this is stable
    cache_control: { type: "ephemeral" as const },
  },
];

const response = await anthropic.messages.create({
  model: process.env.CLAUDE_MODEL!,
  max_tokens: 800,
  tools,
  system: [
    {
      type: "text",
      text: NEARBY_VIBES_SYSTEM_PROMPT,
      cache_control: { type: "ephemeral" },
    },
  ],
  messages: [
    {
      role: "user",
      content: "Find me a casual dinner spot nearby that is open now.",
    },
  ],
});

console.log(response.usage);

The critical thing to log was the usage field, especially the cache-specific tokens:

type CacheUsage = {
  input_tokens: number;
  cache_creation_input_tokens: number;
  cache_read_input_tokens: number;
  output_tokens: number;
};

Those fields let you verify that caching is actually working, rather than assuming it is.

Making Cache Hits Visible

Once you add logging, you can see immediately when a deploy accidentally breaks cache reuse:

function logClaudeUsage(route: string, usage: CacheUsage) {
  logger.info({
    route,
    inputTokens:         usage.input_tokens,
    cacheCreationTokens: usage.cache_creation_input_tokens,
    cacheReadTokens:     usage.cache_read_input_tokens,
    outputTokens:        usage.output_tokens,
    cacheHitRate: usage.cache_read_input_tokens > 0
      ? (usage.cache_read_input_tokens /
         (usage.cache_read_input_tokens + usage.cache_creation_input_tokens))
      : 0,
  }, "claude_usage");
}

A sudden drop in cacheReadTokens after a deployment usually means a tool definition changed, a timestamp got injected into the cached prefix, or dynamic content moved into the wrong section of the prompt.

The Gotchas

Minimum cache length. If the cacheable prefix is too short, it will not be cached even if you mark it with cache_control. The stable prefix needs to be substantial to justify the cache write cost.

Exact matching. The cached prefix has to be byte-identical on every request. If you inject timestamps, request IDs, randomized examples, or any user-specific text into the cached section, each request creates a new cache entry instead of reading the existing one.

Tool churn during development. During active development I was constantly tweaking tool descriptions and schemas. Every tweak invalidated the prefix. Fine while building, but the stable agent contract needs to actually be stable in production before caching delivers real value.

Concurrent request bursts. A cache entry is not available until the first response has started writing it. If you fire a burst of identical requests simultaneously, they may all miss the cache on the first wave.

How the Mental Model Changed

Before caching, the optimization question was: "How small can I make every prompt?"

After caching, the question became: "What parts of this prompt are stable enough to become infrastructure?"

That shift made the agent design cleaner in ways that went beyond cost. Tool definitions became more deliberate. The system prompt became less of a dumping ground. Dynamic context — user location, current session history, real-time context — moved later in the request where it belonged.

The architecture that works well with prompt caching is also just a better-structured agent. The two things reinforce each other.

Takeaway

Prompt caching is not a magic discount button. It rewards stable prompt architecture. For Nearby Vibes, the win came from caching the parts of the agent that behaved like product infrastructure — tools, policies, examples, and stable instructions. The parts that changed per user stayed uncached. That split made the product cheaper to run and easier to reason about.
