Tooling Lessons From Claude Tool Use

Mwiko Sinyangwe · Nearby Vibes · Build notes

The most important part of a tool-using agent is not the model. It is the contract between the model and the tools.

In Nearby Vibes, Claude was not generating text. It was deciding when to search nearby venues, retrieve user memory, check availability, rank options, and trigger follow-up workflows. That meant tools had to be designed like a product API, not like random helper functions bolted on after the fact.

Tool Tiers

I ended up thinking about tools in tiers based on consequence and cost.

Tier 1 — Safe, read-only, cheap:

  • Retrieve user memory
  • Search nearby venues
  • Get cuisine or category suggestions
  • Inspect current location context

Tier 2 — Safe, but more expensive or specific:

  • Fetch detailed venue data
  • Check opening hours
  • Get menu details
  • Run ranking over candidates

Tier 3 — State-changing or user-facing:

  • Save a user memory
  • Schedule a proactive nudge
  • Send a Telegram message
  • Create a nightly BI summary

The model did not get unlimited access across all tiers. The more consequential the tool, the more validation the application performed before executing it. Tier 3 tools often required an explicit check against user preferences and recent action history before firing.

Tool Definitions

The best tool definitions were boring and explicit.

const searchNearbyPlaces = {
  name: "search_nearby_places",
  description:
    "Search for nearby restaurants, cafes, bars, and activities " +
    "that match the user's current request. " +
    "Use when the user wants to find a specific type of venue or food.",
  input_schema: {
    type: "object",
    properties: {
      query: {
        type: "string",
        description:
          "Short search phrase describing what the user wants. " +
          "Examples: 'vegetarian ramen', 'quiet wine bar', 'late night burgers'.",
      },
      lat:  { type: "number", description: "User's current latitude." },
      lng:  { type: "number", description: "User's current longitude." },
      radius_meters: {
        type: "integer",
        minimum: 100,
        maximum: 5000,
        description: "Search radius in meters. Default to 1500 if not specified.",
      },
      open_now: {
        type: "boolean",
        description: "If true, only return venues that are currently open.",
      },
      max_results: {
        type: "integer",
        minimum: 1,
        maximum: 20,
        description: "Maximum venues to return. Default to 5.",
      },
    },
    required: ["query", "lat", "lng", "radius_meters"],
  },
};

I learned to avoid clever names. search_nearby_places beats discover_vibes every time. The model chooses tools far more reliably when names describe the action plainly. Description length matters too — a short first sentence plus specific examples worked consistently better than a long generic description.

Tool Results

Tool result shape matters as much as tool input shape.

Bad result — model has nothing to reason with:

{
  "results": ["Little Sesame, 0.4 miles, good reviews"]
}

Better result — model can explain tradeoffs:

{
  "results": [
    {
      "id": "venue_123",
      "name": "Little Sesame",
      "distance_meters": 640,
      "walk_minutes": 8,
      "open_now": true,
      "price_level": 2,
      "categories": ["Mediterranean", "Vegetarian-friendly"],
      "why_it_matches": [
        "Has strong vegetarian options",
        "Within walking distance",
        "Casual dinner atmosphere"
      ],
      "warnings": ["Several dishes contain nuts"]
    }
  ]
}

The second version gives the model something to reason with. It can explain the match, mention the warnings, and compare options without inventing details. The why_it_matches field was particularly useful — it let the application inject ranking logic that the model could then surface in natural language without needing to re-derive it.

The Agent Loop

async function runToolAgent(messages: Message[]) {
  let currentMessages = messages;

  for (let step = 0; step < 5; step++) {
    const response = await claude.messages.create({
      model:    process.env.CLAUDE_MODEL!,
      max_tokens: 1000,
      system:   NEARBY_VIBES_SYSTEM_PROMPT,
      tools:    nearbyTools,
      messages: currentMessages,
    });

    const toolCalls = extractToolCalls(response);
    if (toolCalls.length === 0) return response;

    const toolResults = await Promise.all(
      toolCalls.map((call) => executeToolSafely(call))
    );

    currentMessages = appendToolResults(currentMessages, response, toolResults);
  }

  throw new Error("Agent exceeded max tool steps — returning partial result");
}

The step < 5 limit is not decorative. Agents need budgets. Without a step budget, a weak query can turn into a slow, expensive loop that burns tokens and leaves the user waiting. The limit forces the system to return something useful rather than searching forever for a perfect answer.

Handling Tool Failures

Tool results need to communicate failure in a way the model can act on:

// Bad — model has no recovery path
throw new Error("Location service unavailable");

// Good — model can respond naturally
return {
  ok: false,
  error_code: "LOCATION_MISSING",
  message: "User location is required before searching nearby places.",
  recoverable: true,
  suggested_action: "Ask the user to share their location.",
};

With a structured failure response, the model recovers naturally: "I can find options for you, but I need your location first." With an unhandled exception, the agent either crashes or produces a confused response.

Logging Tool Calls as Product Events

Tool calls are product events. They should be logged with the same care as any other user action:

logger.info({
  userId,
  sessionId,
  toolName:    call.name,
  arguments:   redactSensitiveFields(call.input),
  durationMs,
  ok:          result.ok,
  errorCode:   result.ok ? null : result.error_code,
}, "agent_tool_call");

Those logs are how you discover that one tool is being chosen for the wrong reasons, that a tool is returning data the model cannot act on, or that latency spikes are coming from a specific external call. The agent loop is observable in the same way any other system is.

The Four Lessons

Make tools narrower than you want at first. A broad handle_user_request tool is easy to build and impossible to control. A narrow search_nearby_places creates a clear contract that both the model and the application can reason about.

Validate tool input server-side. The model proposes arguments. The application owns execution. If a radius must be under 5 km, enforce it in code. If a tool requires user consent, check consent outside the model.

Return structured failures. The model should be able to recover from any tool failure. If your tool can only throw exceptions, the agent cannot recover gracefully.

Log tool calls as product events. Observability is not optional. The only way to understand and improve an agent in production is to have a clear record of what it decided, what tools it called, and what came back.

Takeaway

Tool use works best when the model is treated as a planner and the application is treated as the executor. Claude can decide that it needs nearby restaurants, user memory, or opening hours. But the application defines what those tools mean, validates their inputs, enforces budgets, and decides which actions are allowed. That split made Nearby Vibes feel capable without making it uncontrollable.
