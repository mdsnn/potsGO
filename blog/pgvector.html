How I Built Agent Memory with pgvector

Mwiko Sinyangwe · Nearby Vibes · Build notes

Agent memory sounds abstract until you have a user come back a week later and expect the product to remember something obvious.

In Nearby Vibes, the problem was not "make the agent conscious." It was much more practical: if someone told the assistant they were vegetarian, hated cilantro, usually ordered for two, or preferred cheap late-night places, I did not want every future conversation to start from zero.

The goal was to make the assistant feel continuous across sessions without turning the database into a pile of unstructured chat logs.

The Problem I Was Actually Solving

Nearby Vibes helps users find food and activity options around them. Preferences matter a lot in that context: dietary constraints, cuisines the user likes or avoids, price sensitivity, distance tolerance, whether they are ordering alone or with friends, places they have already tried, and conversational hints like "I want something cozy, not loud."

Some of these are clean structured fields. If a user says "I am vegan," that becomes a profile attribute. But most preferences arrive messily:

Last time the ramen place was good, but I want something less salty and closer to home.

That sentence contains several useful future signals. It does not fit neatly into a single column. This is where pgvector helped — I could store small memory records as text, embed them, and retrieve the most relevant ones when a new request came in.

The Memory Schema

I treated memory as a first-class table, not a side effect of chat history.

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE user_memories (
  id              bigserial PRIMARY KEY,
  user_id         uuid NOT NULL,
  memory_type     text NOT NULL,
  content         text NOT NULL,
  source_message_id uuid,
  confidence      numeric DEFAULT 0.8,
  created_at      timestamptz DEFAULT now(),
  updated_at      timestamptz DEFAULT now(),
  embedding       vector(1536)
);

CREATE INDEX user_memories_user_id_idx
  ON user_memories (user_id);

CREATE INDEX user_memories_embedding_hnsw_idx
  ON user_memories
  USING hnsw (embedding vector_cosine_ops);

I separated memory_type because not all memories behave the same way. A durable preference like "does not eat pork" needs to be treated differently from a soft signal like "was in the mood for sushi last Friday."

Example records in practice:

memory_type: dietary_preference
content: User avoids pork and prefers vegetarian options when available.

memory_type: cuisine_preference
content: User likes ramen, Korean fried chicken, and casual Japanese spots.

memory_type: negative_preference
content: User disliked overly salty ramen and prefers lighter broths.

memory_type: location_preference
content: User usually wants recommendations within a 15-minute walk.

The key decision was to embed the memory statement, not the full chat turn. Full chat turns are noisy. A distilled memory sentence is stable and much easier to retrieve accurately.

Writing Memories

The first version wrote every message into memory. Retrieval got worse almost immediately.

Instead I added a small extraction step after meaningful turns. The model decided whether a conversation turn contained durable preference information. If it did, it returned normalized memory candidates:

type MemoryCandidate = {
  type:
    | "dietary_preference"
    | "cuisine_preference"
    | "negative_preference"
    | "location_preference"
    | "ordering_pattern";
  content: string;
  confidence: number;
};

The application then embedded only the content string and saved it:

async function saveMemory(db, userId: string, memory: MemoryCandidate) {
  const embedding = await embed(memory.content);

  await db.query(
    `INSERT INTO user_memories
       (user_id, memory_type, content, confidence, embedding)
     VALUES ($1, $2, $3, $4, $5)`,
    [userId, memory.type, memory.content,
     memory.confidence, `[${embedding.join(",")}]`]
  );
}

The boring detail that mattered: memories need to be short, declarative, and phrased as reusable facts. "User liked it" is useless six weeks later. "User liked the vegan dumplings at X because they were spicy but not greasy" is actionable.

Reading Memories

At request time, I embedded the user's current intent and pulled back the closest memories for that user using pgvector's cosine distance operator <=>:

SELECT
  id,
  memory_type,
  content,
  confidence,
  1 - (embedding <=> $1::vector) AS similarity
FROM user_memories
WHERE user_id = $2
ORDER BY embedding <=> $1::vector
LIMIT 8;

In practice, I rarely used vector similarity alone. I combined it with type filters so that dietary constraints were always included regardless of similarity score:

SELECT
  id,
  memory_type,
  content,
  confidence,
  1 - (embedding <=> $1::vector) AS similarity
FROM user_memories
WHERE user_id = $2
  AND memory_type IN (
    'dietary_preference',
    'negative_preference',
    'cuisine_preference'
  )
ORDER BY embedding <=> $1::vector
LIMIT 6;

That gave the agent a compact memory block that went into the prompt:

Relevant user memories:
- User avoids pork and prefers vegetarian options when available.
- User disliked overly salty ramen and prefers lighter broths.
- User usually wants recommendations within a 15-minute walk.

The prompt instructed the model to treat memories as useful context, not absolute rules, and to ask if there was a conflict with the current request.

The Query Pattern That Stuck

The most reliable pattern was a two-stage query: cast a wide net first, then filter by confidence and recency.

WITH candidates AS (
  SELECT
    id,
    memory_type,
    content,
    confidence,
    created_at,
    1 - (embedding <=> $1::vector) AS similarity
  FROM user_memories
  WHERE user_id = $2
  ORDER BY embedding <=> $1::vector
  LIMIT 20
)
SELECT *
FROM candidates
WHERE similarity > 0.72
ORDER BY
  confidence DESC,
  similarity DESC,
  created_at DESC
LIMIT 6;

This kept memory small enough to be useful. The agent did not need a biography. It needed the few facts that would actually change the next recommendation.

What Surprised Me

More memory made the agent worse. When I retrieved too many memories, the assistant became overfit to stale context. A user who ordered sushi twice would get sushi suggestions forever. A one-off "I'm craving burgers" could accidentally become a personality trait.

Memory extraction mattered more than embedding quality. A clean memory sentence embedded with an average model consistently outperformed a raw chat transcript embedded with a better model.

Deletion had to be designed early. Users change. Preferences expire. "I am avoiding gluten this week" should not persist for months with the same weight as "I have a peanut allergy." Confidence decay and explicit user correction were both necessary.

What Did Not Work

Treating conversation history as memory was easy to implement and bad in production. Chat turns were too verbose, retrieval matched on incidental words, and old temporary moods resurfaced as if they were permanent preferences.

Retrieving memories by similarity alone also failed for hard constraints. If a user has a dietary restriction, it should not compete against "likes tacos" in vector space. Hard constraints belong in structured profile fields. Vector memory handles the fuzzy preferences that do not fit neatly into columns.

The better architecture was hybrid: structured fields for dietary requirements and hard constraints, vector memory for fuzzy preferences, recency and confidence decay to prevent stale memories from dominating.

Takeaway

pgvector worked well because it kept memory close to the rest of the product data — no separate vector database, no second auth model, no extra operational surface. But the database was not the hard part. The hard part was deciding what deserved to become memory. Agent memory is less about storing everything and more about forgetting on purpose.
