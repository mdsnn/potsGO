Proactive Autonomous Agents in Nearby Vibes

Mwiko Sinyangwe · Nearby Vibes · Build notes

Most agents are reactive. A user asks a question, the agent responds, and the loop ends.

The more interesting version is proactive: the system notices something, decides it is worth acting on, and reaches out or prepares work before the user asks. That is a fundamentally different product — and a fundamentally harder engineering problem.

Nearby Vibes started as a conversational product. But the parts that felt most agentic were not chat responses. They were scheduled jobs, Telegram nudges, and nightly BI workflows running in the background.

Reactive vs Proactive

A reactive agent waits:

User: "Find me dinner nearby." → Agent searches, ranks, replies.

A proactive agent has a standing job:

It is Friday at 5:30pm. This user often asks for casual dinner options around this time. The weather is bad. Their usual spot is closed tonight. Prepare three alternatives and send a light nudge if they have opted in.

That second system is much harder because it has to answer questions the user never asked: Is this useful right now? Is the confidence high enough to act on? Has the user opted in? How recently did we nudge them? What should happen if the recommendation turns out to be wrong?

The Architecture

The proactive layer had three main parts: scheduled jobs, decision logic, and delivery tools. APScheduler handled recurring work. The application handled judgment.

from apscheduler.schedulers.asyncio import AsyncIOScheduler

scheduler = AsyncIOScheduler(timezone="UTC")

scheduler.add_job(
    run_evening_nudge_scan,
    trigger="cron",
    hour="16-20",
    minute="*/15",
    id="evening_nudge_scan",
    replace_existing=True,
)

scheduler.add_job(
    run_nightly_bi_summary,
    trigger="cron",
    hour=2,
    minute=0,
    id="nightly_bi_summary",
    replace_existing=True,
)

scheduler.start()

The scheduler should not contain product intelligence. Its job is to wake up the right workflow at the right time. The workflow decides whether anything deserves to happen.

Telegram Nudges

Nudges were the clearest proactive feature. The system looked for situations where a message might be genuinely useful — user opted in, local time is reasonable, there is a relevant condition (weather, closure, new venue nearby, pattern match), and the user has not been nudged too recently.

async def run_evening_nudge_scan():
    users = await load_users_eligible_for_nudges()

    for user in users:
        context = await build_nudge_context(user)
        decision = await should_send_nudge(user, context)

        if not decision.should_send:
            await log_skipped_nudge(user.id, decision.reason)
            continue

        message = await draft_nudge_message(user, context)
        await send_telegram_message(user.telegram_chat_id, message)
        await record_nudge_sent(user.id, decision.reason)

The critical function is should_send_nudge. That is where product restraint lives:

async def should_send_nudge(user, context) -> NudgeDecision:
    if not user.telegram_opted_in:
        return NudgeDecision(False, "not_opted_in")

    if context.local_hour < 10 or context.local_hour > 21:
        return NudgeDecision(False, "quiet_hours")

    if context.nudges_sent_last_7_days >= 2:
        return NudgeDecision(False, "frequency_cap")

    if not context.has_strong_recommendation:
        return NudgeDecision(False, "low_confidence")

    if context.days_since_last_app_open > 14:
        return NudgeDecision(False, "user_inactive")

    return NudgeDecision(True, "timely_recommendation")

The agent can draft the message. It should not decide whether sending it is appropriate. That decision belongs in explicit, auditable application logic — not in a model call.

Nightly BI

Nightly BI was proactive in a different direction. It did not message users. It prepared operational intelligence for the product team.

The workflow pulled the day's events, aggregated usage, tool calls, recommendation outcomes, and failure patterns, asked the model to summarize notable signals, and stored the result:

async def run_nightly_bi_summary():
    events  = await load_events_for_previous_day()
    metrics = aggregate_product_metrics(events)
    summary = await summarize_metrics_with_model(metrics)

    await save_bi_summary(
        date=metrics.date,
        metrics=metrics,
        summary=summary,
    )

The model was useful here because product telemetry is noisy. A good summary could surface things like:

Search quality degraded for late-night requests — most
returned venues were already closed at time of query.

Tool timeout rate increased after 8pm local time.

Users asked follow-up questions about vegetarian options
more frequently than the previous week.

Recommend: check venue hours data freshness and review
evening search radius defaults.

That is not just reporting. It is the system preparing tomorrow's work.

The Autonomy Ladder

Autonomy is not a single switch. I think of it as a ladder:

  • Level 1: System drafts something. A human reviews and sends it.
  • Level 2: System sends low-risk messages under strict, explicit rules.
  • Level 3: System takes reversible actions — saves a memory, creates a reminder, updates a preference.
  • Level 4: System takes consequential actions — books, purchases, contacts other parties on the user's behalf.

Nearby Vibes stayed mostly at Levels 2 and 3. That was intentional. Food recommendations, nudges, and summaries are useful, but they should not create surprise obligations for users. Level 4 autonomy needs much stronger trust infrastructure — explicit delegation, reversible actions, clear audit trails, and easy correction paths.

Where Autonomous Agents Break Down

Acting without enough signal. A proactive agent needs a meaningful reason to act, not just an available opportunity. Sending a nudge because it is 6pm and the user has not opened the app is noise. Sending one because their usual spot is unexpectedly closed and there is a strong alternative nearby is useful.

Stale context. The context that drives proactive decisions decays. A user's dinner-time patterns from two months ago may not reflect their life today. Proactive decisions need recency checks, not just preference checks.

Frequency without restraint. The first nudge feels helpful. The fifth one this week feels like an app that does not respect attention. Frequency caps are not optional — they are what makes proactive features feel considerate rather than aggressive.

No explanation trail. If a proactive action causes a problem — wrong recommendation, bad timing, intrusive message — the user (and the product team) needs to be able to understand why it happened. Every proactive action should have a logged reason that a human can read.

The Primitives That Make It Work

The technical implementation is not exotic. A scheduler wakes up jobs. Jobs collect context. The model classifies, summarizes, or drafts. Tools deliver the result. What makes it trustworthy is the guardrail layer:

  • Explicit opt-in for any user-facing proactive feature
  • Quiet hours that cannot be overridden by any model decision
  • Frequency caps per user per feature per time window
  • Confidence thresholds before any message is drafted
  • Audit logs with human-readable reasons for every action taken and skipped
  • Easy opt-out that the system respects immediately
Takeaway

The future of agents is not better chat boxes. It is systems that can watch, prepare, and occasionally act at the right moment. But proactive does not mean pushy. The best autonomous agents understand timing, permission, and silence. The most powerful action an agent can take is sometimes deciding to do nothing.
