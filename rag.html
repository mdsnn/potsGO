LangChain: What I Used, What I Did Not, and Why

Mwiko Sinyangwe · Nearby Vibes · Build notes

LangChain is one of those tools that people tend to discuss as a belief system. You are either supposed to love it because it gives you abstractions for LLM apps, or reject it because it hides too much.

My take is more boring: LangChain is useful when its abstractions match the shape of your problem. It gets in the way when you are still discovering that shape.

For Nearby Vibes, I evaluated LangChain early, borrowed some of its vocabulary, and built the production agent without it. That is not a verdict against LangChain. It is a verdict about where the agent loop sat in the product.

What LangChain Gets Right

LangChain is genuinely good at getting a prototype moving quickly. If you need to wire together a model, prompt templates, tools, retrieval, memory, and output parsing, it gives you a vocabulary and a component set. That matters when you are still learning what an LLM application even needs.

The value is not only the code. It is the mental structure. Concepts like tools, retrievers, chains, agents, and memory are useful labels regardless of whether you end up using the library. LangChain helped push the whole community toward shared vocabulary for AI system design, and that vocabulary is worth having even if you go your own way on the implementation.

I would reach for LangChain faster on a project where the integration surface is broad and standard: chat over documents, common vector stores, interchangeable model providers, standard loaders and retrievers. The abstractions fit that shape well.

Where It Started to Feel Heavy

The more product-specific Nearby Vibes became, the more I cared about the exact behavior between steps. I wanted direct control over how tools were selected, how memory was retrieved, how many tokens each section consumed, how errors surfaced, how the agent recovered from bad tool output, and how telemetry mapped to product events.

A general-purpose orchestration framework can become another layer to debug when product behavior is the thing you are actually trying to tune. The issue was not that LangChain could not do these things. The issue was that I was spending more time understanding the abstraction than understanding the product behavior underneath it.

The Agent Loop I Actually Wanted

For Nearby Vibes, the core loop was simple enough to own directly:

async function runNearbyAgent(input: AgentInput): Promise<AgentResult> {
  const memory = await retrieveRelevantMemory(input.userId, input.message);

  const response = await claude.messages.create({
    model:    process.env.CLAUDE_MODEL!,
    max_tokens: 1000,
    system:   buildSystemPrompt({ memory }),
    tools:    nearbyTools,
    messages: input.messages,
  });

  if (response.stop_reason === "tool_use") {
    const toolResults = await runRequestedTools(response.content);
    return continueWithToolResults(input, response, toolResults);
  }

  return formatAgentResult(response);
}

That is not a lot of code. More importantly, every line maps directly to product behavior. When something went wrong, I could answer exactly where it broke: memory retrieval, tool selection, tool execution, or final formatting. That debugging path mattered more to me than having a more elegant framework graph.

The Abstractions I Did Borrow

Even without using LangChain directly, I ended up thinking in its vocabulary:

  • Retrievers — the memory and venue retrieval layer in Nearby Vibes is essentially a retriever, just implemented directly in Postgres and TypeScript
  • Tools — the concept of strongly typed tool contracts with defined inputs and structured outputs came from the broader ecosystem LangChain helped popularize
  • Structured outputs — forcing model outputs into typed shapes, not free-form text
  • Evaluation traces — logging agent steps as structured events so you can replay and debug what the agent decided

The ideas are valuable. The question is always whether the implementation layer adds more than it costs.

The Rule I Use Now

Use frameworks for commodity complexity. Own the code where product behavior lives.

For Nearby Vibes, tool orchestration, memory, ranking, and prompt caching were product behavior. I wanted those pieces sitting directly in the application code where I could see them, test them, and change them without navigating a framework's abstraction layer.

Integrating with a model provider, loading documents, chunking text, connecting to a vector store — these are commodity complexity. LangChain can handle that well, and using it there would have been reasonable.

The agent reasoning loop, the tool contract, the memory retrieval strategy — these were product decisions. I kept those close.

When I Would Choose LangChain

I would reach for LangChain when:

  • The project needs to work across multiple model providers and I want to swap them easily
  • I am building retrieval over documents and the retrieval pipeline is mostly standard
  • I need to prototype quickly and proving the concept matters more than owning every line
  • The team is already familiar with it and the onboarding cost is already paid

I would build without it when:

  • The agent loop is the product — the way the agent reasons and acts is the differentiator
  • I need precise control over prompt structure, caching, and token budgets
  • The debugging surface needs to be as thin as possible
  • The team needs to understand every step without consulting external documentation
Takeaway

The best LangChain decision is not yes or no. It is deciding which parts of your AI system are generic enough to outsource to a framework and which parts are specific enough that you should own them. For Nearby Vibes, the agent loop pushed firmly into the second category.
